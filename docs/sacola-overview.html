Estado atual

A plataforma sai do estágio embrionário com os primitivos de identidade e organização prontos. Em produção (sacola.pop.dev.br) funcionam hoje:

  • Auth completa — cadastro self-service, email de ativação (Resend), login com sessão cookie httpOnly (6h), reenvio de ativação. Anti-enumeração por timing (dummy bcrypt) e por shape de resposta.
  • Empresas multi-tenant — qualquer usuário ativado cria uma empresa e vira dono. Mais empresas por usuário, mais usuários por empresa.
  • Roles em duas camadas — 3 genéricos (owner/admin/member) + 4 específicos do domínio Sacola (gerente/vendedor/separador/entregador). Mesmo motor de permissão escala pra outros POPs.
  • Convites por email — TTL de 7 dias, vale como verificação de email (o convidado pula a etapa de ativação se cria conta inline).
  • Gestão de membros — promover/rebaixar, remover, sair, transferir propriedade atômica, reenviar convite (rotaciona o token).
  • Auditoria — log append-only de ações de gerenciamento, visível para admin+ em /app/[slug]/auditoria.

Suite de testes: 159 testes (29 suítes) cobrindo unit + integration end-to-end via mailpit. CI roda Jest + Prettier + ESLint + TypeScript + Commitlint + Vercel build preview por PR. Deploy via push em main.

Tutorial — primeiro contato

O passo-a-passo abaixo cobre o caminho dourado do produto hoje. Tudo se passa em sacola.pop.dev.br.

1Criar conta

  1. Em /, clica em Criar conta.
  2. Preenche username (3-32 alfanuméricos / underscore), email e senha (12+ chars com pelo menos um caractere especial). Submete.
  3. Você cai em /cadastro/aguardando-ativacao?email=… e recebe um email do remetente Sacola <noreply@contato.pop.dev.br> com um link que expira em 15 minutos. Se não chegou, use o botão Reenviar email de ativação.
  4. Clica no link → cai em /cadastro/ativar/[token] → o frontend chama a API automaticamente → vê Conta ativada!
  5. De volta em /login, entra com email + senha. Cookie de sessão é setado e você é redirecionado pra /app.
Por baixo dos panos: a senha vai por bcryptjs com 14 rounds em produção (e 1 round em dev/test, pra não travar a suite). No login, o backend faz compare mesmo quando o email não existe (com um hash dummy), pra timing e shape ficarem idênticos — não dá pra enumerar quem é cadastrado.

2Criar uma empresa

  1. Em /app, clica em Criar empresa.
  2. Digita o nome. O slug (parte da URL) é sugerido em tempo real a partir do nome — "Mercado do João" vira mercado-do-joao. Pode editar; se já existe, a API tenta sufixos numéricos (-2, -3…) automaticamente.
  3. Submete → cai em /app/[slug] como Dono, com a equipe listada (você sozinho por enquanto).

3Convidar membros

  1. No dashboard da empresa, clica em Equipe (ou diretamente /app/[slug]/equipe).
  2. Convidar → preenche email + escolhe função no dropdown (Gerente, Membro, Vendedor, Separador, Entregador). Submete.
  3. Resend envia o email pro convidado com um link /convite/[token] (TTL 7 dias). O convite aparece em "Convites pendentes" com botões Reenviar (rotaciona o token) e Revogar.
  4. Convidado clica no link:
    • Já tem conta com o email convidado → vê "Aceitar como email@…" e entra direto.
    • Não tem conta → vê o form inline (username + senha). O email já está provado pelo clique no link, então a conta é criada já ativada — pula completamente o fluxo de email de ativação.
    • Logado com email diferente → bloqueio explícito com mensagem "este convite é para outro email; faça login com a conta convidada".

4Gerenciar equipe e auditoria

  • Na Equipe: cada linha de membro tem um dropdown de função (PATCH na hora) e botão Remover. Owner não aparece nos controles — protegido contra remoção/troca pela API.
  • Em Auditoria (botão no dashboard, admin+): linha do tempo reverso com todos os eventos de gerenciamento (criação, convite, aceite, troca de função, remoção, renomear, transferir propriedade). Cada entrada mostra quem fez, o que aconteceu e quando.

5Sair, transferir ou excluir

  • Membros não-owner: em /app/[slug]/configuracoes, card "Sair desta empresa" → confirm dialog → você perde acesso imediato.
  • Owner: precisa transferir antes. Card "Transferir propriedade" → escolhe um membro no dropdown → digita o username pra confirmar → submit. Você vira Gerente, o destinatário vira Dono. Depois pode usar "Sair desta empresa" normalmente.
  • Excluir a empresa inteira: só owner. Card "Excluir empresa" → digita o slug pra confirmar → DELETE remove a empresa + memberships + convites em sequência (compensatória, sem FK).
A transferência é atômica via compensação: o owner é demovido pra admin primeiro, depois o destinatário é promovido. Se o segundo update falhar, o owner é re-promovido — nunca ficamos com zero owners (o estado mais catastrófico, porque ninguém mais poderia deletar nem transferir a empresa).

Features documentadas

Autenticação

Recurso Como funciona
Cadastro self-service POST /api/v1/users valida username (regex), email (regex + max 254 chars), senha (12+ chars com especial). Cria registro com features = ["read:activation_token"] (unactivatedUser) e dispara email com link.
Ativação PATCH /api/v1/activations/[token]. Token é 64-hex (32 bytes random), TTL 15min, single-use. Em sucesso, features do user trocam pra activatedUser.
Reenvio de ativação POST /api/v1/activations { email }. Sempre 202. Internamente: noop se email não existe ou já está ativado (anti-enum), senão deleta tokens pendentes + manda novo.
Login POST /api/v1/sessions. Cookie sacola_session_id (httpOnly, SameSite=Lax, Secure em prod), TTL 6h. Sessão é DB-backed (token aleatório de 96-hex armazenado em sessions).
Anti-enumeração Wrong-password e ghost-email retornam mesmo body ("Email ou senha inválidos.") e mesmo timing (dummy bcrypt no caminho do ghost para igualar a duração da comparação real).
Conta não ativada Se as credenciais batem mas o user está unactivated, retorna 401 com mensagem clara ("Sua conta ainda não foi ativada") — só após a senha ser verificada (não vaza a quem não conhece a senha).

Multi-tenancy (empresas)

Toda entidade de negócio é scoped por empresa. Um usuário pertence a N empresas via N rows na tabela memberships; o role da linha é a fonte da verdade pras permissões dentro daquela empresa.

Operação Endpoint Permissão
Criar empresa POST /api/v1/companies create:company (qualquer ativado)
Listar minhas empresas GET /api/v1/companies logado
Ver detalhes GET /api/v1/companies/[slug] read:company (member+)
Editar nome/slug PATCH /api/v1/companies/[slug] update:company (admin+)
Excluir DELETE /api/v1/companies/[slug] delete:company (owner)
Transferir propriedade POST /api/v1/companies/[slug]/transfer-ownership delete:company (owner)

Convites

Operação Endpoint Permissão
Enviar convite POST /api/v1/companies/[slug]/invitations create:invitation (admin+)
Listar pendentes GET /api/v1/companies/[slug]/invitations read:invitation (admin+)
Revogar DELETE /api/v1/companies/[slug]/invitations/[id] delete:invitation (admin+)
Reenviar (rotaciona token) POST /api/v1/companies/[slug]/invitations/[id]/resend create:invitation (admin+)
Landing pública GET /api/v1/invitations/[token] público
Aceitar POST /api/v1/invitations/[token]/accept público (cria sessão)
Convite é prova de email. O endpoint de aceitar tem dois ramos: com sessão existente, exige email da sessão == email do convite; sem sessão, recebe { username, password }, cria conta já ativada e atribui membership. Em ambos os casos, retorna cookie de sessão.

Gestão de membros

Operação Endpoint Notas
Listar membros GET .../members member+. Inclui username; nunca email.
Trocar role PATCH .../members/[user_id] admin+. Owner é blindado; promoção pra "owner" via PATCH é proibida.
Remover outro DELETE .../members/[user_id] admin+. Não pode remover owner, não pode remover a si mesmo (caminho separado abaixo).
Sair da empresa DELETE .../members/me qualquer membro não-owner; idempotente.

Auditoria

Cada ação mutativa loga uma linha em audit_log (append-only, indexado por company_id, created_at DESC). Listing em GET /api/v1/companies/[slug]/audit-log (admin+), surfaceado na UI em /app/[slug]/auditoria.

Ações registradas:

company.created          company.updated
ownership.transferred
invitation.created       invitation.revoked       invitation.resent
member.joined            member.role_changed      member.removed     member.left

O actor_id é resolvido na camada HTTP (a sessão sabe quem é); o log NÃO mora nos modelos pra eles continuarem reaproveitáveis em CLI/orchestrator/testes sem contaminar a auditoria com eventos fantasmas. Se a inserção do log falha, o usuário ainda vê sucesso (a mutação já ocorreu); o erro vai pro console.error.

Arquitetura e padrões

Stack

  • Runtime: Node 24, TypeScript
  • Framework: Next.js 16 (App Router, sem src/)
  • DB: Postgres 16 (Neon em prod, Docker compose em dev) acessado via pg direto — sem ORM, SQL na mão
  • Migrações: node-pg-migrate, forward-only, sem .down
  • Email: nodemailer sobre SMTP — Resend em prod (sa-east-1), mailpit em dev
  • Auth: sessão DB-backed (não JWT), bcryptjs pra senhas (14 rounds em prod)
  • UI: shadcn primitives copiados em components/ui/, Tailwind 4, next-themes pra dark/light global
  • Testes: Jest, real Next.js dev server + Postgres + mailpit (sem mocks de DB/email)
  • Deploy: Vercel (push em main → produção), Neon-Vercel integration cria DB branches por preview

Layout do repositório

app/                Next.js App Router pages + API routes
  (authed)/         route group com gate de sessão
  api/v1/           endpoints REST
components/         shadcn primitives + componentes compartilhados
infra/              database, mailer, errors, controller, scripts, migrations
lib/                helpers puros (sem DB) — usáveis em client/server
models/             camada de dados (1 arquivo por agregado)
tests/              integration tests (testam contra dev server + DB real)
docs/               este documento

Modelo de permissões em 2 camadas

O check isAuthorized(user, feature, scope?) decide qual camada se aplica pela presença da feature em cada catálogo.

Camada 1 — Global Camada 2 — Scoped por empresa
Onde users.features (array) memberships.role → ROLE_PERMISSIONS[role]
Pra quê login, perfil, criar empresa membros, convites, auditoria, e (futuro) produtos/pedidos/estoque
Features create:session · read:session · update:user · read:user:self · read:status · create:company read:company · update:company · delete:company · read/update/delete:member · read/create/delete:invitation · read:audit_log

Catálogo de roles (lib/roles.ts):

Role Tipo Permissões hoje
owner genérico tudo + delete:company
admin genérico tudo de management; não pode excluir
member genérico read-only
gerente Sacola == admin
vendedor Sacola == member
separador Sacola == member
entregador Sacola == member
Os 4 roles do domínio Sacola começam alinhados com os genéricos e vão divergir em Semana 2-3, quando os endpoints de produto/pedido/estoque chegarem. A estrutura já está em pé pra esse momento (Open/Closed): adicionar um create:order em vendedor é uma linha em lib/roles.ts, sem mexer em ninguém mais.

Schema atual

users                       (id, username, email, password, features[], timestamps)
sessions                    (id, token, user_id, expires_at, timestamps)
user_activation_tokens      (id, token, user_id, used_at, expires_at, timestamps)
companies                   (id, name, slug UNIQUE, timestamps)
memberships                 (id, user_id, company_id, role, timestamps)
                              UNIQUE (user_id, company_id) + indexes em ambas FKs
invitations                 (id, company_id, email, role, token UNIQUE, invited_by,
                              expires_at, accepted_at, timestamps)
audit_log                   (id, company_id, actor_id, action, target_type, target_id,
                              metadata jsonb, created_at)
                              INDEX (company_id, created_at DESC)

Sem FK constraints — convenção do projeto (espelhando automanews, repositório de referência). Cascades são em código (ex: deleteCompanyById chama deleteMembershipsByCompany antes do DELETE da company).

Como rodar localmente

git clone https://github.com/joaohenriquecosta/sacola.git
cd sacola
cp .env.example .env.development     # já vem com defaults dev-ready
npm ci
npm run dev                          # sobe Postgres + mailpit + Next, roda migrations

Em http://localhost:3000 a aplicação roda igual à produção, com email capturado em http://localhost:8025 (mailpit web UI). Testes: 

npm test                # full suite (sobe Docker + Next, espera prontidão)
npm run test:watch      # watch mode
npm run lint && npm run format:check && npm run typecheck

Detalhes de deploy em DEPLOY.md. Convenções de migration em infra/migrations.md. PRs sempre passam por CI completo + Vercel preview antes de merge em main.
