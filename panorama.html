1. O que é

Sacola é um sistema operacional para hortifruti com pedidos via WhatsApp, separação distribuída em mini-estoques e entrega por motoboys informais. MVP com cliente real: 3 vendedores, 3 separadores, 3 entregadores, ~50 pedidos/dia.

Dor principal: hoje tudo passa por WhatsApp + planilha e dá muito trabalho de conferir (pedido, separação, entrega, recebimento). O sistema mata a redigitação e faz tudo "bater" — daí o nome.

Papéis:

  • Gerente — cadastros, abastecimento, dashboard, confirmação de pagamentos
  • Vendedor (3) — recebe pedido por WhatsApp, lança no Sacola, registra pagamento
  • Separador (3) — vê fila, separa item a item pesando, finaliza
  • Entregador (3) — vê fila de prontos, marca em rota / entregue / voltou

6. Cronograma — 4 semanas

Semana Entrega O que faz
1 Fundação + Auth POST /api/v1/sessions funcionando; CI verde; CLI cria primeiro gerente.
2 Cadastros + Estoque Gerente cadastra produtos, vincula a separador, lança abastecimento, lê saldo — tudo via API.
3 Pedido (core) Vendedor cria → separador finaliza → estoque baixa → cancelamento devolve. Transações atômicas.
4 Pagamento + Frontend + Deploy API de pagamento; frontend mobile-first usando shadcn; deploy Vercel+Neon; cliente real usando.

7. Eventos em tempo real (decisão)

SSE (Server-Sent Events) + polling no servidor a cada 10 segundos.

  1. Tabela orders ganha updated_at timestamptz com trigger que reescreve a cada UPDATE.
  2. Função watchChanges({since}) em models/order.ts é um async generator: roda SELECT orders WHERE updated_at > $since a cada 10s e emite os novos.
  3. Rota app/api/v1/events/orders/route.ts consome o gerador e empurra pra cliente via text/event-stream.
  4. Cliente usa new EventSource("/api/v1/events/orders"). Reconnect é nativo do browser.

Por quê SSE e não WebSocket: SSE é HTTP puro, unidirecional (que basta — o cliente fala pelo POST/PATCH normais), funciona em qualquer Vercel Function, reconnect automático. WebSocket seria overkill aqui.

Por quê polling e não LISTEN/NOTIFY puro: 10s de delay é invisível pro fluxo (separação leva minutos). E a abstração watchChanges() deixa trocar pra LISTEN/NOTIFY depois sem mexer em mais nada.

9. Pendências / decisões abertas

  • Formato exato da delivery_tag — sugestão atual: iniciais-N (ex: MS-1). Confirmar com cliente.
  • Política de logs em action_logs — registrar: login, criação de usuário, troca de senha, criar/cancelar pedido, ajustes de estoque, confirmar pagamento. Falta decidir granularidade.
  • Limite do alerta de saldo baixo — sugestão MVP: ≤ 10% do estoque ou ≤ 1 unidade. A confirmar em uso real.
  • Primeiro deploy — Vercel + Neon. Conta Neon: pendente criar.
  • Próximo passo de código — escolher entre: (a) git commit do estado atual, (b) Husky+commitlint+CI, (c) Jest setup, (d) primeira migration.
